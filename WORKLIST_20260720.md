## Session 3 — User ID re-serialisation + orphan cleanup + email-change fixes

### User ID re-serialisation

**Problem:** App-registered users had IDs `99000100–99000123` (AUTO_INCREMENT pre-set to 99000001 at DB creation), while admin-created users via `newIdGive` were in the `1111120–1111155` range. Two disjoint ranges.

**Migration** (PHP/PDO script, `SET FOREIGN_KEY_CHECKS=0`, single transaction):
- `99000100 → 1111156`, `99000101 → 1111157`, …, `99000123 → 1111179` (24 users)
- Updated all 86 user-referencing columns across 66 tables including `personal_access_tokens.tokenable_id`, `imie_histories`, `comments`, `host_data`, `agencies`, `followers`, `visitors`, `day_times`, `short_video_*`, and more
- `users.id` and `users.phone` updated for each remapped row
- `AUTO_INCREMENT` reset to **1111180** — new registrations continue from there

Users table is now fully sequential: **1111120 → 1111179**, next = **1111180**.

### Phone field race-condition fix (all 3 AuthControllers: `Api`, `V4`, `V5`)

- **Old:** `$lastId = User::latest('id')->value('id'); ... 'phone' => $lastId + 1` — computed BEFORE the INSERT; any concurrent registration or AUTO_INCREMENT gap causes `phone ≠ id`
- **New:** Remove `$lastId` line; `'phone' => 0` in `User::create()`; then `$new_user->phone = $new_user->id; $new_user->save()` — phone always equals actual DB-assigned id
- Backups: `.bak_phone_20260720` on both nodes; php-fpm reloaded

### DB data repair (one-time SQL)

`UPDATE users SET phone = id WHERE phone IS NULL OR phone != id` — fixed 17 rows:

- User 99000100 (now 1111156): `phone=1111156` was already the new ID so was correct post-remap

## Session 4 — UserEmailChangeStore bug fix + custom user creation

### `UserEmailChangeStore` two bugs fixed (`app/Http/Controllers/Admin/UserController.php`)

Backup: `.bak_emailfix_20260720` on both nodes.

**Bug 1 — New ID create always returned "User not found"**

Root cause: the `else` branch (ID not found → create new user) had a `return back()->with('User not found')` as its **first statement**, making all create code below it dead/unreachable. Any admin attempt to create a custom ID via the "ID Swap" form always returned an error.

**Bug 2 — Email swap didn't guard against same-user**

Root cause: `if ($is_email_exist)` ran the swap path even when the submitted email already belonged to the same user whose ID was entered. This caused a redundant self-swap (email briefly changed to `id@queenlive.com` then restored), risking a unique-constraint collision.

**Fixed behaviour:**

| Scenario | Result |
|---|---|
| ID exists, email free | Updates email → success |
| ID exists, email belongs to another user | Swaps that user to `id@queenlive.com`, sets new email → success |
| ID exists, email already on same user | No-op save → success |
| ID free, email free | Creates new user: id, phone=id, name="Custom ID", pass=123456 → success |
| ID free, email taken | Error: "email exists on user X, use swap" |

php-fpm reloaded on both nodes after patch.
